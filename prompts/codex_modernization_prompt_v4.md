# 25. Дополнительная policy для структурных stacked-диаграмм

## 25.1. Подпись итоговой суммы по столбцу

Если столбец содержит **2 и более структурных элемента**, сверху обязательно выводить **итоговую сумму по всему столбцу**.

Правило:
- считать total = сумма всех сегментов столбца;
- показывать total над столбцом;
- форматировать в тех же единицах, что и ось Y;
- если график показывает объем размещения, подпись должна быть в **млрд рублей**;
- если столбец состоит из одного элемента, итоговая подпись допустима, но не обязательна.

Пример:
```text
8 046,9
```
или
```text
8 046,9 млрд руб.
```
Выбрать единый стандарт в рамках графика.

## 25.2. Подписи сегментов

Для внутренних сегментов:
- показывать подпись значения внутри сегмента, если сегмент достаточно велик;
- если сегмент слишком мал, не перегружать график, а показывать значение через hover;
- допустимо использовать порог:
  - сегмент подписывается, если его доля в столбце >= 5%;
  - или если его абсолютное значение превышает минимальный визуальный порог.

## 25.3. Hover для stacked-диаграмм

- период;

# 26. Глобальная palette policy: запрет на соседство синего и желтого в структурных диаграммах

Во всех диаграммах проекта, особенно stacked bars и stacked areas, **не использовать синий и желтый как соседние структурные цвета**.

Причина:
- визуальный дисбаланс;
- плохая воспринимаемость в stacked-структуре;
- риск визуального "разрыва" и некомфортного соседства при восприятии сегментов;
- нестабильная читаемость подписей на фоне светло-желтого сегмента рядом с насыщенным синим.

## 26.1. Обязательное правило

Для `scripts/palette.py` добавить правило:

- запрещенная соседняя пара:
  - `blue + yellow`
  - `yellow + blue`

Это правило применять для:
- maturity structure;
- format structure;
- monthly structure charts;
- Sankey node palettes, если узлы находятся рядом в логике одной структуры;
- dashboard stacked charts.

## 26.2. Рекомендуемая палитра для сроков обращения

Использовать более устойчивую палитру, например:

- долгосрочные: темно-фиолетовый / пурпурный;
- среднесрочные: бирюзовый / teal;
- краткосрочные: коралловый / терракотовый / мягкий красный;
- requires_review: серый.

То есть:
- не использовать комбинацию `фиолетовый + синий + желтый`;
- заменить ее на комбинацию с более равномерной контрастностью.

Пример допустимого распределения:
```text
Долгосрочные  -> #7B2CBF
Среднесрочные -> #2A9D8F
Краткосрочные -> #E76F51
requires_review -> #9AA0A6
```

Допускается использовать другую палитру, если:
- она контрастна;
- русские легенды сохранены;
- нет соседства синего и желтого;
- подписи читаемы.

## 26.3. Проверка palette compliance

Нужно ввести проверку, что для stacked structure charts:

# 29. Критическая доработка: broken yield_boxplot_by_ofz_type_year_cumulative

Текущая проблема:
график `yield_boxplot_by_ofz_type_year_cumulative_2026-01-01_retrospective_5.html` остается методологически и визуально сломанным даже после предыдущих правок.

Наблюдаемые дефекты:
1. В faceted long-retrospective режиме boxplot фактически не отображается как boxplot:
   - коробки и усы либо отсутствуют, либо визуально схлопнуты;
   - точки выстроены в один вертикальный кластер;
   - периоды на оси X фактически не разведены по годам внутри панели.
2. Подписи статистик (`min`, `max`, `median`, `n`, `upper/lower fence`) выводятся в одном месте и не привязаны к конкретному году / boxplot.
3. Внутри каждой панели по типу ОФЗ годы не читаются как отдельные категории.
4. Для длинной ретроспективы логика построения short-mode и long-mode по сути смешалась, из-за чего график перестал выполнять задачу сравнительного анализа распределения доходности.

## 29.1. Обязательное исправление архитектуры boxplot

Нужно привести `yield_boxplot_by_ofz_type` к двум четким режимам:

### short mode
Если число периодов `<= 3`:
- X = `ofz_type`;
- color = `period`;
- один boxplot на каждую пару (`ofz_type`, `period`);
- допустим grouped mode.

### long mode
Если число периодов `> 3`:
- использовать `facet_col = ofz_type`;
- внутри каждой панели X = `report_period_display_label` или `report_year`;
- X обязан быть категориальным и упорядоченным слева направо по возрастанию периода;
- один boxplot = один период внутри конкретного `ofz_type`;
- точки размещений (strip/scatter) могут накладываться поверх boxplot, но с jitter и по соответствующей оси X этого периода.

Запрещено:
- схлопывать все периоды в одну категорию внутри панели;
- рисовать long-retrospective режим как фактически одну вертикальную полосу точек;
- размещать статистические подписи вне связи с конкретным boxplot периода.

## 29.2. Подписи статистик для long-mode

Для длинной ретроспективы:

- основные подписи на графике:
  - `median`;
  - `n`;
  - при необходимости `max` и `min`, но только рядом с соответствующим boxplot и только если они не создают перегрузку.

- `q1`, `q3`, `upper fence`, `lower fence`, полные экстремумы и прочие детальные статистики:
  - переносить в hover;
  - сохранять в chart export / analytical export.

Правило:
- каждая подпись должна быть якорена к соответствующему X-периоду и Y-значению;
- нельзя собирать подписи всех лет в середине панели.

## 29.3. Экспорт статистик boxplot

Для boxplot необходимо сформировать или стабилизировать export-таблицу статистик:
- `report_period_start`;
- `report_period_display_label`;
- `report_period_order`;
- `ofz_type`;
- `n`;
- `min`;
- `q1`;
- `median`;
- `q3`;
- `max`;
- `lower_fence`;
- `upper_fence`;
- `has_outliers`;
- `outliers_count`.

Эта таблица должна использоваться и для проверки качества графика.

## 29.4. Специальная QA-проверка для boxplot

`html_chart_qa.py` должен уметь проверять:

1. что для long-mode внутри каждой панели по `ofz_type` есть несколько distinct X-категорий по годам/периодам;
2. что boxplot traces действительно построены;
3. что подписи `median/n` не схлопнуты в одну точку;
4. что short-mode и long-mode дают разную архитектуру построения;
5. что порядок периодов слева направо корректный;
6. что ось Y называется:
   `Доходность, % годовых`.
